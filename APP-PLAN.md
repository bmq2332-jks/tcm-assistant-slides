## 1. 市場同類應用分析

### 1.1 參考對象總覽

| App | 平台 | 技術棧 | API Key 自備 | 開源 | 核心功能 |
|-----|------|--------|:---:|:---:|----------|
| **中濟經絡穴位** | Android | 原生 NDK | ❌ 內建 | ❌ | 3D 經絡穴位 + AI 面/舌/手診 + 影片課程 |
| **倪海厦漢唐中醫診斷助手** | Android | Flutter + Dart + SQLite | ✅ 完全離線 (無需 API) | ✅ MIT | 六經辨證診斷 + 經方處方 + 子午流注 |
| **傷寒論刷題寶典** | Android | 未公開 (推 Kotlin) | ✅ OpenRouter Key | ✅ MIT | LLM 無限刷題 + AI 繪圖 + 弱點分析 |
| **說點啥 BiBi Keyboard** | Android | Kotlin + MD3 + OkHttp | ✅ 使用者自填 | ✅ GPL | 語音辨識 (10+ ASR) + LLM 後處理 |
| **輕聲細語 Whisper To Input** | Android | 未公開 | ✅ OpenAI/自建 | ✅ 開源 | OpenAI Whisper STT 輸入法 |
| **愛看舌 / AI 舌診** | 多平台 | 雲端 API | ❌ 內建 | ❌ | 舌象辨識 + 體質分析 |
| **有方中醫** | iOS | Swift + 雲端 AI | ❌ 內建 | ❌ | AI 舌診面診 + 養生管家 |

### 1.2 關鍵市場洞察

1. **API Key 自備模式已成主流趨勢** — 說點啥、傷寒論刷題寶典 都採用使用者自行輸入 API Key 的方式，優點是：
   - 開發者無需負擔 API 成本
   - 使用者可選擇自己信任的服務商
   - 避免隱私爭議（資料直接發送到使用者指定的端點）

2. **離線優先 + AI 雲端混合架構** — 倪海廈診斷助手完全離線運行（無需網路），傷寒論刷題寶典則是離線題庫 + 雲端 LLM。本 App 建議採用 **本地中醫知識庫 + 雲端 AI 診斷** 混合模式。

3. **開源中醫 AI App 正在崛起** — GitHub 上出現多個中醫相關開源專案（倪海廈助手、傷寒論刷題寶典），代表社群對此需求旺盛。

4. **中醫 AI 診斷 API 供應商漸趨成熟** — 舌診、面診、體質辨識等 AI API 已有多家服務商（百度靈醫、貝葉斯、脈景、小善心等），具備商業化水準。

## 2. 產品定位與核心價值

### 2.1 產品願景

> **「你的中醫口袋助手 — 自備 API Key，資料在你手，診斷 AI 幫你走。」**

不同於中濟經絡穴位這類封閉式 App，本 App 讓使用者：
- 自行選擇 & 輸入 AI API Key（OpenAI / Google Gemini / 自建端點）
- 所有診斷資料僅發送到使用者指定的 API 端點
- 本地中醫知識庫完全離線可用
- 無訂閱制、無廣告、完全透明

### 2.2 目標用戶

| 用戶畫像 | 需求場景 | 技術能力 |
|----------|----------|:--------:|
| 中醫學生/實習醫師 | 學習經絡穴位、練習辨證論治 | 低-中 |
| 中醫自學者/養生愛好者 | 日常症狀查詢、養生建議 | 低 |
| 中醫師/針灸師 | 快速取穴參考、AI 輔助診斷 | 中-高 |
| AI / 開源技術愛好者 | 自建模型串接、研究整合 | 高 |

### 2.3 與中濟經絡穴位的差異化

| 維度 | 中濟經絡穴位 | 本 App |
|------|------------|--------|
| **API 模式** | 內建固定服務 | **使用者自備 API Key** |
| **AI 診斷** | 內建（不可選） | **多供應商可選** (OpenAI / Gemini / 自建) |
| **隱私** | 資料由開發者處理 | **資料直達使用者指定端點** |
| **3D 模型** | 內建 3D 人體模型 | 初期用 2D 示意圖 + WebView |
| **費用** | 免費 + 內購 | **完全免費 + 自備 Key** |
| **開源** | ❌ | ✅ **MIT 開源** |

## 6. 技術實現方案

### 6.1 技術選型

| 層級 | 技術方案 | 選擇理由 |
|------|----------|----------|
| **UI 框架** | **Flutter 3.x + Material Design 3** | 一套程式碼 Android/iOS 都跑，開發效率高；倪海廈助手使用 Flutter 驗證可行；熱重載快速迭代 |
| **語言** | Dart 3.x | Flutter 官方語言，sound null safety |
| **狀態管理** | Riverpod 2.x 或 BLoC | 輕量、型別安全、測試友善 |
| **本地資料庫** | SQLite (sqflite 或 drift) | 離線知識庫存放經絡、穴位、方劑資料 |
| **本地快取** | Hive / Isar | 診斷記錄、使用者設定、Key 加密儲存 |
| **網路請求** | Dio + 攔截器 | 發送到使用者指定的 API 端點；支援串流回應 |
| **加密儲存** | flutter_secure_storage | API Key 用 iOS Keychain / Android EncryptedSharedPreferences 保護 |
| **相機/相簿** | camera + image_picker | 拍攝舌象、面診照片 |
| **語音輸入** | speech_to_text (可選) | 輔助症狀輸入 |
| **主題** | Material You (Monet) 自適應 | Android 12+ 動態色彩 |
| **資料視覺化** | fl_chart | 體質雷達圖、診斷趨勢圖 |
| **WebView** | webview_flutter | 未來嵌入 3D 人體模型 |

#### 支援的 AI 服務商

#### API 請求封裝 (簡化範例)

```dart
class AIService {
  final String baseUrl;
  final String apiKey;
  final String model;

  /// 舌診分析：傳送圖片，回傳診斷結果
  Future<TongueDiagnosis> analyzeTongue(Uint8List image) async {
    final base64Image = base64Encode(image);
    final response = await _postChatCompletion([
      ChatMessage(
        role: 'system',
        content: tongueAnalysisPrompt,  // 精心設計的中醫舌診 System Prompt
      ),
      ChatMessage(
        role: 'user',
        content: [
          ContentText(text: '請分析這張舌象照片，輸出 JSON 格式診斷報告。'),
          ContentImage(base64: base64Image),
        ],
      ),
    ]);
    return TongueDiagnosis.fromJson(response);
  }

  /// 症狀問答：串流 streaming 回覆
  Stream<String> chatStream(List<ChatMessage> messages) {
    return _postChatCompletionStream(messages);
  }
}
```

#### 關鍵：System Prompt 工程

AI 診斷品質的關鍵不在模型，而在 **System Prompt**。例如舌診 prompt 需包含：

```
你是一位精通中醫舌診的專家。請根據使用者上傳的舌象照片，輸出以下 JSON 格式：

{
  "tongue_body": "舌質描述（淡紅/紅/暗紅/紫/淡白）",
  "tongue_coating": "苔色與苔質（薄白/黃/膩/剝落）",
  "tongue_shape": "舌體形態（胖大/齒痕/裂紋/正常）",
  "pattern": "辨證結果（如：脾虛濕盛）",
  "analysis": "簡要分析（30字內）",
  "diet_therapy": ["建議食療1", "建議食療2"],
  "acupoints": ["建議穴位1", "建議穴位2"],
  "lifestyle": ["生活建議1", "生活建議2"]
}

注意：
- 只輸出 JSON，不要額外文字
- 基於中醫辨證論治理論
- 若無法判斷，請誠實標示不確定
```

> 類似的 Prompt 也適用於面診、體質辨識、症狀問答。

## 7. API Key 管理策略

### 7.1 安全性設計

```
┌──────────────┐
│  使用者輸入   │ ← 首次啟動強制設定
│  API Key     │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  flutter_    │ ← 加密儲存 (Android: EncryptedSharedPreferences)
│  secure_     │    iOS: Keychain Services
│  storage     │    第三方無法直接讀取
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  Memory      │ ← App 運行期間僅存在記憶體
│  (Provider)  │    不會寫入 log 或 crash report
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  HTTP Header │ ← 每次請求從記憶體讀取
│  Authorization│    不持久化寫入 SharedPreferences
└──────────────┘
```

### 7.2 Key 管理功能

| 功能 | 實作方式 |
|------|----------|
| 輸入 Key | 遮罩顯示 (sk-...xxxx)，可切換顯示/隱藏 |
| 測試連線 | 發送輕量請求 (列出模型)，確認 token 有效 |
| 更換服務商 | 一鍵切換 OpenAI / Gemini / 自訂，清除舊 Key |
| 清除 Key | 設定頁面「清除所有 Key」按鈕 (需二次確認) |
| 多 Key 管理 | (進階) 支援不同服務設定不同 Key |
| 資料流向提示 | 診斷時顯示「資料將發送至 api.openai.com」 |

### 7.3 隱私透明度

- 每個診斷畫面上方顯示：「🔒 資料直送 [服務商網址]，開發者無法存取」
- App 本身 **不收集任何匿名分析資料**
- 診斷記錄儲存在本機，不經由開發者伺服器
